**课程名称**：编译原理与技术实践
**实验日期**：2026年01月13日

实践亮点：
1. 在 符号管理 模块中，针对不同阶段的访问特性采用了最佳实践的数据结构：**红黑树 (std::map)**管理关键字(1)；**字典树 (Trie)** 与 **哈希表** 的双向映射架构，实现了 Token 的 $O(L)$ 快速前缀匹配与单字节别名映射，与反向映射还原完整 Token，兼顾了 **高性能与易用性** (2)； **直接寻址表 (Array)** 优化状态转移效率(3)； **哈希表 (std::unordered_map)** 实现符号表的 $O(1)$ 均摊查找与插入。(1、2、3、4)
2. 在 LR分析器 中，采用了 **自动化表生成**（`maker.cpp`），实现了First/Follow 集推导、LR 分析表的自动生成，并进一步实现了 **分析表的DFA 状态机代码自动生成**（直接导出为 C++ 头文件），消除了运行时建表的开销。(3)
3. 在 错误处理环节 中，设计了 **多层次错误恢复机制**：语法层采用了恐慌模式、虚拟插入与前瞻验证(2、3)；语义层实现了 类型检查与运行时错误检测。(2、3、4)
4. 在 词法与语法分析 阶段，采用了 **确定性有限自动机 (DFA)** 与 **预测分析表驱动架构**，分别实现了基于状态迁移的复杂 Token 识别（如科学计数法）和无回溯的高效语法解析。(1、2)
5. 在 语法树可视化 模块中，结合了 Graphviz DOT 语言，开发了 `Visualizer.h` 模块，实现了 AST的自动DOT代码生成与图片导出，直观对比LL与LR的建树差异。(2、3)
6. 在 语义分析 阶段，采用了 **解释器模式与中间代码生成并行** 的策略，在生成四元式（IR）的同时维护符号表状态，实现了变量值的实时计算与类型自动提升，并支持符号表快照（CSV）导出功能。(4)
7. 在 错误定位环节 中，设计了 **精确行号追踪机制**，通过记录上一个成功匹配的 Token 位置，解决了在存在空行或跨行语句时错误报告行号等常见问题。(2、3)
8. 在 测试与验证 阶段，编写了 **自动化测试流水线**（`run_tests.py`），集成了测试用例的批量执行、可视化生成、输出结果保存到文件等功能。(1、2、3、4)
9. 在 文件读取与缓冲 模块中，实现了 **双缓冲区** 机制，通过类封装底层 I/O，有效减少了系统调用次数，并提供了字符级的安全过滤功能，增强了词法分析器对非标准输入的鲁棒性。(1、4)
10. 在 项目架构设计 中，采用了 **高内聚低耦合的模块化设计**：LR 分析表构建采用了 **生产者-消费者模式**（`maker.cpp` 生成，解析器消费），实现了生成逻辑与解析逻辑的解耦；**可视化模块** (`Visualizer`) 独立封装，支持跨模块（LL/LR）复用；各分析阶段通过标准化接口连接，构建了灵活可扩展的编译流水线。(1、2、3、4)
